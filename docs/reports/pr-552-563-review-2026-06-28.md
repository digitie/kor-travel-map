## HIGH — MOIS feature-update runner가 dataset_key별로 중복 feature를 양산 (#561 + #562)

**근본원인은 하나.** `feature_update_runner.py`의 MOIS spec이 4개 dataset_key를 모두 같은 경로로 dispatch한다:

```
# feature_update_runner.py:648-655
dataset_keys=frozenset({MOIS_BULK, MOIS_HISTORY, MOIS_CLOSED, MOIS_DETAIL})
run=run_feature_place_mois_licenses
resources=_mois_resources

# _mois_resources (397-404) — dataset_key로 분기 없음
"mois_license_records": fetch_mois_license_records(settings),   # 항상 동일
"mois_dataset_key":     scope.dataset_key or MOIS_BULK_DATASET_KEY,  # 라벨만 바뀜

# provider_fetchers.py:304-340 — open 레코드만 스트리밍
yield from mois_db.iter_open_place_records(session, service_slugs=PROMOTED...)
```

`mois_dataset_key`는 `license_records_to_bundles(dataset_key=...)` → `make_feature_id(source_type=f"{PROVIDER}:{dataset_key}", ...)`로만 흘러간다. `make_feature_id`가 source_type을 해시에 넣으므로:

- **#561**: `/ops/providers`에서 `mois_license_features_history` 또는 `_closed`(둘 다 catalog상 `is_feature_load=True` + `is_refreshable=True`) refresh를 누르면 → **현재 영업중(open) 레코드 전체**가 `dataset_key=...history/...closed` namespace의 **별도 feature_id**로 적재됨. 즉 "폐업/이력" 데이터셋이 활성 업소로 채워짐. (게다가 `license_records_to_bundles`는 `is_open != True`를 스킵하므로 closed 레코드는 애초에 들어올 수도 없음.)
- **#562**: `mois_license_detail`을 `is_refreshable=True`로 올려 동일 spec에 추가 → detail refresh가 **open bulk 42 PROMOTED slug 전체를 새 namespace로 복제**. detail fetcher가 없으니 enrichment는 0이고, `mng_no` 기준 merge도 없어 bulk feature와 합쳐지지 않음.

**임팩트**: 이전엔 "지원하지 않는 provider/dataset"으로 **시끄럽게 실패**하던 요청이, 이제 **조용히 성공하며 잘못된 데이터를 쓴다**. 운영자 클릭 1회로 MOIS place feature가 최대 2~3배로 불어나고 지도에 중복 마커/스토리지 증가. catalog에 검증이 없어 request 생성 시 차단되지 않음.

> 보강 관찰(cross-cut): #562의 detail 조회 endpoint(`routers/mois_detail.py`)는 **어떤 프런트/소비자에서도 호출되지 않는 dead endpoint**다 (`git grep "debug/mois-license"` → 생성된 types.ts만 매치). 즉 detail 기능의 두 product surface(중복 양산 runner + dead endpoint) 모두 end-to-end로 도달 불가.

**권장**: history/closed/detail은 실제 history/closed/detail 소스를 읽는 전용 fetcher가 생기기 전까지 `is_refreshable=False`(+ `is_feature_load=False`)로 되돌리거나, runner에서 명시적 실패로 라우팅. detail은 ADR-016 secondary source(`is_primary_source=False`)로 기존 bulk feature_id에 붙이는 방식이라야 enrichment가 됨.

## MEDIUM

**M1 · #558 — dedup count 쿼리가 매 요청마다 전체 큐를 materialize (perf).**
`_DEDUP_REVIEW_COUNT_SQL`은 `WITH reviews AS MATERIALIZED (...)` + `feature.features` 2회 INNER JOIN + source_links/records LATERAL 2회를, **필터가 없어도** 필터된 전체 큐에 대해 계산 후 `count(*)`. 페이지 클릭·debounce 필터 타이핑마다 실행됨. 또 list가 keyset→OFFSET으로 바뀌며 deep page에서 keyset 인덱스 바운드를 잃음. 회귀 테스트(`test_t212d...keyset_indexes`)는 offset 0의 LIST SQL만 EXPLAIN하고 count SQL·deep offset은 검증 안 함. (admin 내부망/저트래픽이라 medium.)
→ 필터 부재 시 join/lateral 없이 `dedup_review_queue`에서 바로 count(`idx_dedup_status_score` 활용), count SQL에 EXPLAIN 단언 추가.

**M2 · #559 — enrichment "detail source 선택"이 audit-only로, 적용 데이터를 바꾸지 않음 (foot-gun).**
다이얼로그의 `정리된 datagokr` / `visitkorea` 선택은 `_reason_with_detail_source`로 `decision_reason` 문자열에 `detail_source=...`만 덧붙일 뿐, accept 경로(`decide_enrichment_review` → `_rebuild_enrichment`)는 항상 저장된 `source_record`로 재구성. 두 선택이 **바이트 동일 결과**. 관리자가 "visitkorea 강제 적용"으로 오해. (cross-cut F3도 동일 확인.)
→ 선택을 apply 경로로 실제 연결하거나, 컨트롤을 "기록용 메모"로 relabel + 툴팁.

**M3 · #561 — data.go.kr curated fileData 월간 스케줄이 4개 중 1개만 갱신.**
`DATAGOKR_FILEDATA_DATASETS`는 4개(seoul_bookstores / gyeonggi_muslim / ansan_world / jeju_local)인데, 단일 asset+단일 월간 schedule이 `datagokr_file_data_dataset_key` resource(기본값 `datagokr_seoul_bookstores`)만 materialize. 나머지 3개는 운영자가 수동 요청해야만 적재됨. 스케줄 설명("curated fileData 월 1회 적재")이 전체 커버를 암시해 사일런트 갭.
→ dataset별 schedule(또는 partitioned/loop job) 분리, 혹은 "기본 1개만, 나머지 on-demand" 명시.

**M4 · #563 — features-write live spec이 같은 PR의 "plan-only" 불변식과 모순되게 무조건 실제 DB write.**
이 PR이 `playwright.live.config.ts`를 "백업/업데이트 요청은 execute=false / dry-run plan-only만 허용"으로 고쳐놓고, 같이 추가한 `admin-features-change-requests-write.live.spec.ts`는 **env 게이트 없이** create→approve(applied)→PATCH→deactivate→DELETE를 실제 타겟에 반영. backups spec은 `E2E_BACKUP_RESTORE_EXECUTE*`로 게이트하는데 이건 무방비. `npm run e2e:live`(문서상 prod는 `E2E_LIVE_ALLOW_PROD=1`)가 실 feature를 생성/삭제.
→ `test.skip(!process.env.E2E_ADMIN_FEATURES_WRITE, ...)`로 게이트하거나 config 주석 수정.

**M5 · #560 — 이번 PR이 세운 "Linux/WSL git 단일 실행" 정책과 모순되는 문서가 남음.**
